请介绍一下优化后的 SmartNotes 项目，它的整体架构和 7 大优化点是什么？
基础 gateway/main.py agent_qa/main.py
▼

参考答案

一句话概括：SmartNotes 是一个面向大学生的智能学习助手，采用微服务架构，通过统一网关 + 多 Agent 模式提供笔记整理、复习计划、智能问答和知识库管理四大功能。优化后实现了用户数据隔离、缓存版本控制、批量写入等 7 大核心改进。

架构分层：

  1. 前端层（Vue3 + Element Plus + Pinia）：赛博朋克暗色主题，支持 Markdown 实时渲染、思考中动画、打字机光标
  2. 网关层（FastAPI Gateway :8000）：统一入口，负责 JWT 认证、Redis 滑动窗口限流、路由转发到下游 Agent
  3. Agent 层：三个独立 FastAPI 服务
    • agent_note :8001 — 笔记整理
    • agent_plan :8002 — 复习计划
    • agent_qa :8003 — 智能问答 + 知识库管理
  4. 数据层：MySQL 8.0（用户/知识库元数据）、Redis 6.2（限流/缓存）、ChromaDB（向量检索）
  5. 模型层：通过 LangChain 调用阿里百炼 qwen3.7-plus，兼容 OpenAI SDK

7 大核心优化点：

  1. 知识库按用户隔离：collection_name 绑定 user_id，格式为 user_{id}_{name}
  2. LLM 缓存 key 包含模型版本号：避免模型升级后缓存失效或返回旧版本结果
  3. ChromaDB 批量写入优化：长文本按段落分块，单次调用批量添加，减少网络往返
  4. 前端 Markdown 实时渲染：使用 marked + DOMPurify 实现安全的 Markdown 渲染
  5. 删除时同步清理 ChromaDB：MySQL 删除记录后，同步删除对应向量数据
  6. 复习计划流式输出修复：修复 chunk.trim() 过滤导致内容丢失的 bug
  7. QA 思考中动画和打字机光标：流式响应前显示跳动圆点动画，输出时显示闪烁光标

加分点

提到"为什么选择微服务架构"——每个 Agent 可以独立扩缩容、独立部署、独立升级模型，比如问答 Agent 需要连接 ChromaDB 而其他不需要。优化后的 7 个点都是针对生产环境的真实痛点。

你的项目中流式输出（Streaming）是怎么实现的？从用户点击到看到第一个字，经过了哪些环节？
中等 QA.vue gateway/main.py qa_chain.py
▼

参考答案

完整链路：

  1. 前端（QA.vue）：使用原生 fetch 发起 POST 请求，通过 response.body.getReader() 获取 ReadableStream，用 TextDecoder 逐块解码，追加到 buffer 实现逐字显示
  2. Gateway（gateway/main.py）：调用 _proxy_stream_request()，使用独立的 httpx.AsyncClient 以 stream=True 模式向下游 Agent 发起请求，通过 StreamingResponse 将流透传给前端
  3. Agent（agent_qa/main.py）：端点 /ask/stream 返回 StreamingResponse，内部调用 qa_chain.answer_stream()
  4. Chain（qa_chain.py）：LangChain 链 self.chain.astream() 异步流式调用 LLM
  5. LLM（llm_service.py）：ChatOpenAI(streaming=True) 启用流式模式，通过 SSE 协议逐 token 返回

加分点

提到 Gateway 的流式透传用了独立 httpx.AsyncClient（生命周期由 stream_generator 管理），而不是复用全局 client，避免了长连接占用全局连接池的问题。同时设置了 X-Accel-Buffering: no 防止 Nginx 缓冲流式响应。

你的 RAG 知识库问答是怎么做的？检索不到相关内容时怎么处理？
中等 qa_chain.py agent_qa/main.py chroma_client.py
▼

参考答案

RAG 流程：

  1. 知识入库：用户通过 /knowledge/add/text 或文件上传接口提交文本，Agent QA 同时写入 ChromaDB（自动 Embedding）和 MySQL knowledge_items 表（元数据镜像）
  2. 语义检索：用户提问时，chroma_client.search() 将问题向量化，在 ChromaDB 中用余弦相似度搜索 Top-5 相关文档
  3. 上下文拼接：_build_context_from_search_results() 将检索结果格式化为 [文档 1 - 来源]\n内容 的结构化文本
  4. LLM 生成：将 context 注入 RAG Prompt 模板，通过 qa_chain.answer_stream() 流式生成回答
  5. 结果回写：问答结果自动保存回 ChromaDB，形成知识闭环

检索不到时的处理：

  • ChromaDB 搜索异常时，except 捕获后将 context 设为空字符串，降级为普通问答
  • QA Chain 中有两套 Prompt：rag_prompt（有知识库上下文）和 normal_prompt（无上下文），根据 context 是否为空自动切换
  • RAG Prompt 中明确要求模型："如果知识库中没有相关内容，请根据你的知识回答，但请注明'此回答未基于知识库内容'"

进阶优化方向

可以提到：混合检索（向量 + BM25 关键词）、Re-ranking 重排序、Query Expansion 查询扩展、HyDE 假设性文档嵌入等 Advanced RAG 技术。

你的项目中 Redis 有哪些用途？限流算法是怎么实现的？
中等 redis_client.py gateway/main.py
▼

参考答案

Redis 的四大用途：

  1. 滑动窗口限流：使用 ZSET（有序集合），key 为 rate_limit:{user_id/ip}，score 为时间戳
  2. LLM 响应缓存：QA 问答结果以 MD5(question+context+model) 为 key 缓存，TTL 3600 秒
  3. Session 管理：支持 session 的 get/set/delete/refresh，TTL 86400 秒
  4. 通用 KV 缓存：cache_get / cache_set / cache_delete 接口

滑动窗口限流算法：

pipe = redis.pipeline()
pipe.zremrangebyscore(key, 0, window_start)  # 1. 清除窗口外的旧记录
pipe.zadd(key, {str(now): now})               # 2. 添加当前请求时间戳
pipe.zcard(key)                                # 3. 统计窗口内请求数
pipe.expire(key, window + 1)                   # 4. 设置过期时间
results = await pipe.execute()
if results[2] > max_requests:                  # 5. 超限则回滚并拒绝
    await redis.zrem(key, str(now))
    return True  # 被限流

加分点

使用 Redis Pipeline 将 4 个操作合并为一次网络往返，减少延迟。限流 key 优先使用 user_id（已登录），未登录则降级为 IP。窗口参数可通过环境变量配置（默认 60 秒 / 100 次）。

你的 JWT 认证方案是怎样的？Token 过期了怎么处理？
基础 jwt_auth.py gateway/main.py
▼

参考答案

认证方案：

  1. 注册：密码用 bcrypt.hashpw() 加密后存入 MySQL，返回 JWT Token
  2. 登录：查询 MySQL 验证用户，bcrypt.checkpw() 校验密码，生成 JWT
  3. Token 结构：Payload 包含 sub（user_id）、username、iat（签发时间）、exp（过期时间，默认 24h）、type: "access"
  4. 验证：JWTAuth.get_current_user() 从 Authorization Header 或 Cookie 获取 Token，解码验证
  5. 传递：Gateway 解析出 user_id 后，通过 X-User-Id Header 传递给下游 Agent

过期处理：

  • 后端：jwt.ExpiredSignatureError 捕获后返回 401 + "Token 已过期"
  • 前端 Axios 拦截器：捕获 401 状态码，清除 localStorage 中的 token，自动跳转到 /login
  • Vue Router beforeEach 守卫：检查 authStore.isLoggedIn，未登录重定向到登录页

你的 LangChain Chain 是怎么设计的？为什么用 prompt | llm | parser 这种写法？
中等 note_chain.py llm_service.py
▼

参考答案

LCEL（LangChain Expression Language）链式写法：

self.chain = self.prompt | self.llm | self.output_parser

这是 LangChain 推荐的 LCEL 语法，使用 | 操作符将 Runnable 组件串联：

  1. self.prompt（ChatPromptTemplate）：将模板变量替换为实际值，生成消息列表
  2. self.llm（ChatOpenAI）：接收消息列表，调用大模型 API 生成回复
  3. self.output_parser（StrOutputParser）：将 LLM 输出的 AIMessage 对象解析为纯文本字符串

为什么用 LCEL：

  • 统一接口：所有组件都实现 Runnable 接口，支持 invoke()、astream()、batch() 等统一方法
  • 自动流式支持：调用 astream() 时，LCEL 自动将流式传递到每个组件，无需手动处理
  • 可组合：可以轻松插入 Retriever、Memory 等中间组件
  • 懒加载：LLM 实例通过单例模式的 @property 懒加载，避免启动时就需要 API Key

加分点

提到 LLMService 使用了单例模式（__new__ + _instance），确保全局只有一个 ChatOpenAI 实例和连接池，减少资源消耗。

优化后的项目相比第一版有哪些核心改进？请逐一说明。
困难 agent_qa/main.py Plan.vue QA.vue
▼

参考答案

1. 知识库按用户隔离（user_id 绑定 collection_name）：

  • 原问题：所有用户共享同一个 default collection，数据互相可见
  • 优化：_get_user_collection(user_id, collection_name) 生成 user_{id}_{name} 格式，确保用户数据隔离
  • 好处：多用户场景下数据安全，避免 A 用户检索到 B 用户的知识库内容

2. LLM 缓存 key 包含模型版本号：

  • 原问题：MD5(question+context) 不包含模型信息，模型升级后缓存可能返回旧版本结果
  • 优化：key_content = f"qa:{settings.LLM_MODEL}:{question}:{context or ''}"
  • 好处：切换模型后缓存自动失效，避免版本不一致问题

3. ChromaDB 批量写入优化：

  • 原问题：逐条调用 collection.add()，网络往返多
  • 优化：长文本按 \n\n 分块，批量生成 IDs 和 metadatas，单次 add_texts() 调用
  • 好处：减少网络往返，提升入库速度

4. 前端 Markdown 实时渲染：

  • 原问题：流式输出是纯文本，Markdown 语法（如 ##、-）直接显示，不美观
  • 优化：引入 marked 解析 Markdown + DOMPurify 做 XSS 过滤，computed 实时渲染
  • 好处：输出即时格式化，支持标题、列表、代码块等，同时防止 XSS 攻击

5. 删除时同步清理 ChromaDB：

  • 原问题：删除 MySQL 记录后，ChromaDB 中向量数据残留，造成"幽灵数据"
  • 优化：delete_knowledge_item() 中先查 MySQL 获取 chroma_id，再调用 collection.delete(ids=[chroma_id])
  • 好处：数据一致性，避免已删除文档仍被检索到

6. 复习计划流式输出修复：

  • 原问题：Note.vue 中使用 if (chunk && chunk.trim()) 过滤空 chunk，导致 Markdown 中的空行、缩进被丢失
  • 优化：Plan.vue 中改为 result.value += chunk（不过滤），保留所有空白字符
  • 好处：Markdown 格式完整保留，列表缩进、代码块换行不再丢失

7. QA 思考中动画和打字机光标：

  • 原问题：流式响应前页面无反馈，用户不知道 AI 是否在处理
  • 优化：QA.vue 中添加 thinking-indicator（三个跳动圆点）和 typing-cursor（闪烁竖线）
  • 好处：提升用户体验，明确告知用户"AI 正在思考"和"正在输出"

LLM 的 Token 是什么？Temperature 参数有什么作用？你在项目中是怎么设置的？
基础 settings.py
▼

参考答案

Token：LLM 处理文本的最小单位，不等于字或词。一个中文词通常 1-2 个 Token，英文一个单词约 1-4 个 Token。API 按 Token 数量计费（输入 + 输出）。

Temperature：控制模型输出的随机性/创造性。

  • 低值（0-0.3）：输出更确定、保守，适合事实性问答、代码生成
  • 中值（0.5-0.7）：平衡创造性和准确性，适合通用对话
  • 高值（0.8-1.0+）：输出更多样、随机，适合创意写作

项目中的设置：

# settings.py
LLM_TEMPERATURE = 0.7     # 通用默认值
LLM_MAX_TOKENS = 4096     # 最大输出 Token 数

# llm_service.py
ChatOpenAI(temperature=0.7, max_tokens=4096, streaming=True)

# retriever.py 中的 RAG 场景
ChatOpenAI(temperature=0.4)  # RAG 问答用更低温度，减少幻觉

加分点

提到 RAG 场景下用低 Temperature（0.4）是因为检索已经提供了上下文，不需要模型"发挥创意"，降低幻觉概率。而笔记整理和计划生成用 0.7，允许更多创造性表达。

什么是 Prompt Engineering？你在项目中用了哪些 Prompt 技巧？
中等 note_chain.py plan_chain.py qa_chain.py
▼

参考答案

Prompt Engineering 是通过精心设计输入提示，引导 LLM 生成更准确输出的技术。

项目中使用的技巧：

  1. Role Prompting（角色扮演）：每个 Chain 都定义了明确的系统角色
    • 笔记整理："你是一位专业的学习笔记整理助手"
    • 复习计划："你是一位专业的学习规划师"
    • 智能问答："你是一个智能学习助手"
  2. 结构化输出约束：在 System Prompt 中明确输出格式要求（Markdown 格式、层级标题、列表等）
  3. 条件 Prompt（策略模式）：QA Chain 根据是否有检索上下文，动态选择 RAG Prompt 或普通 Prompt
  4. 变量模板：使用 ChatPromptTemplate 的 {content}、{style_hint}、{preferences} 等占位符，运行时动态填充

可扩展的技巧

还可以加入：Few-shot Prompting（给示例）、Chain-of-Thought（思维链）、负面提示（告诉模型不要做什么）。比如在笔记整理中加入一个优秀笔记的示例作为 Few-shot。

什么是 LLM 的幻觉（Hallucination）？你在项目中如何缓解这个问题？
中等
▼

参考答案

幻觉定义：LLM 生成看似合理但实际上不正确、不存在或与事实不符的内容。

项目中的缓解措施：

  1. RAG 检索增强：让 LLM 基于检索到的真实文档回答，而非仅凭训练数据"编造"
  2. Prompt 约束：RAG Prompt 中明确要求"如果知识库中没有相关内容，请注明'此回答未基于知识库内容'"
  3. 低 Temperature：RAG 场景用 0.4（retriever.py），降低随机性
  4. 知识闭环：问答结果保存回 ChromaDB，后续相似问题可以检索到历史回答，减少重复幻觉

注意

当前项目没有实现"引用来源标注"功能——即在回答中标注信息来自哪个文档。这是可以优化的方向。

什么是 Embedding？ChromaDB 在你的项目中是怎么用的？
中等 chroma_client.py
▼

参考答案

Embedding（嵌入）：将文本转换为高维向量（如 1536 维），使语义相似的文本在向量空间中距离更近。通过余弦相似度等度量方式计算文本间的语义相关性。

ChromaDB 在项目中的使用：

  1. 连接方式：使用 chromadb.HttpClient 连接独立的 ChromaDB 服务（Docker 部署），而非嵌入式模式
  2. 集合管理：get_or_create_collection()，使用余弦相似度（hnsw:space: cosine）
  3. 写入：add_texts() 传入文本列表 + 元数据 + ID，ChromaDB 自动调用 Embedding 模型进行向量化
  4. 检索：collection.query() 将查询文本向量化，返回 Top-K 最相似的文档及距离分数

加分点

提到 ChromaDB 使用 HNSW（Hierarchical Navigable Small World）索引算法，这是一种近似最近邻（ANN）算法，在百万级向量下仍能保持毫秒级检索速度。

  • document_loader.py 中已准备 RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
  • 优化后的 agent_qa/main.py 中知识入库时已实现分块：按 \n\n 分割段落，批量写入 ChromaDB
  • 每个段落独立生成 ID 和 metadata（包含 paragraph_index 和 user_id）

加分点

提到批量写入优化——分块后不是逐条 add，而是批量生成所有 IDs 和 metadatas，单次 chroma_client.add_texts() 调用，减少网络往返。

如何评估 RAG 系统的效果？你会用什么指标？
困难
▼

参考答案

检索质量指标：

  • Recall@K：Top-K 结果中包含相关文档的比例
  • MRR（Mean Reciprocal Rank）：第一个相关文档排名的倒数均值
  • NDCG：考虑位置权重的归一化折损累计增益

生成质量指标：

  • Faithfulness（忠实度）：回答是否基于检索到的文档（RAGAS 框架）
  • Answer Relevance（回答相关性）：回答是否切题
  • Context Precision（上下文精度）：检索到的文档中有多少是真正有用的

工程指标：

  • 端到端延迟（检索 + 生成总时间）
  • 缓存命中率（项目中的 Redis 缓存）
  • 用户满意度（点赞/点踩反馈）

工具推荐

可以使用 RAGAS（RAG Assessment）框架自动化评估，或 LangSmith 进行链路追踪和评估。

你的项目中知识库数据同时存了 ChromaDB 和 MySQL，为什么要双写？删除时如何保证一致性？
中等 agent_qa/main.py database.py
▼

参考答案

双写设计的原因：

存储职责优势
ChromaDB向量检索语义搜索、相似度计算
MySQL knowledge_items元数据管理关系查询、用户隔离、删除管理
  1. 用户隔离：MySQL 通过 user_id 外键实现数据权限控制，ChromaDB 的元数据过滤能力较弱
  2. 删除操作：先从 MySQL 删除记录（/knowledge/{item_id}），再根据 chroma_id 从 ChromaDB 清理
  3. 数据一致性：MySQL 作为 Source of Truth，ChromaDB 作为检索索引
  4. 关系查询：可以按用户、时间、文件名等维度查询知识库条目列表

删除时的一致性保证（优化后）：

# agent_qa/main.py 中的删除逻辑
result = await session.execute(
    select(KnowledgeItem).where(
        KnowledgeItem.id == item_id,
        KnowledgeItem.user_id == x_user_id,
    )
)
item = result.scalar_one_or_none()

# 同步从 ChromaDB 删除对应的向量数据
collection = chroma_client.get_or_create_collection(item.collection_name)
collection.delete(ids=[item.chroma_id])

await session.delete(item)
await session.commit()

潜在问题

当前双写没有分布式事务保障——如果 ChromaDB 删除成功但 MySQL commit 失败，数据会不一致。生产环境应该考虑事务性 Outbox 模式或最终一致性重试机制。

什么是向量数据库？和传统数据库有什么区别？为什么 RAG 需要它？
基础
▼

参考答案

对比维度传统数据库（MySQL）向量数据库（ChromaDB）
数据类型结构化数据（数字、文本、日期）高维向量（浮点数数组）
查询方式精确匹配、SQL 条件查询近似最近邻搜索（ANN）
索引结构B+ Tree、Hash IndexHNSW、IVF、PQ
相似度度量等于/大于/小于余弦相似度、欧氏距离、内积
适用场景事务处理、关系查询语义搜索、推荐系统、RAG

RAG 为什么需要向量数据库：

  1. LLM 无法直接搜索海量文本，需要先"检索"相关文档
  2. 关键词搜索（LIKE '%关键词%'）无法理解语义，"手机"搜不到"移动电话"
  3. 向量数据库通过 Embedding 将文本转为向量，实现语义级别的相似度搜索
  4. 支持百万级文档的毫秒级检索（ANN 算法）

使用方式：

# 异步引擎（主流程使用）
async_engine = create_async_engine(
    settings.mysql_url,         # mysql+aiomysql://
    pool_size=10,               # 连接池大小
    max_overflow=20,            # 溢出连接数

    pool_recycle=3600,           # 连接回收时间

# 同步引擎（Alembic 迁移等工具使用）

为什么用异步：

  1. FastAPI 是异步框架，同步数据库操作会阻塞事件循环，导致所有请求排队等待
  2. aiomysql 驱动在等待数据库响应时释放事件循环，可以处理其他请求
  3. AI 应用中数据库查询通常不是瓶颈（LLM API 调用才是），但异步可以避免不必要的阻塞

ORM 模型：使用 declarative_base() 定义了 User 和 KnowledgeItem 两个模型，通过 ForeignKey 建立关联，ondelete="CASCADE" 实现级联删除。

Docker Compose 中服务依赖是怎么管理的？为什么 Gateway 要 depends_on 所有 Agent？
基础 docker-compose.yml
▼

参考答案

依赖管理：

agent_note:
  depends_on:
    mysql: { condition: service_healthy }  # 等待 MySQL 健康检查通过
    redis: { condition: service_started }   # 等待 Redis 启动即可

agent_qa:
  depends_on:
    mysql: { condition: service_healthy }
    redis: { condition: service_started }
    chromadb: { condition: service_started }  # 额外依赖 ChromaDB

gateway:
  depends_on:
    agent_note: { condition: service_started }
    agent_plan: { condition: service_started }
    agent_qa: { condition: service_started }

为什么 Gateway 依赖所有 Agent：

  1. Gateway 的职责是转发请求到下游 Agent，如果 Agent 没启动，转发会失败
  2. 使用 service_started 而非 service_healthy，因为 Agent 没有定义健康检查端点（实际上有 /health 但未在 Compose 中配置）
  3. MySQL 使用 service_healthy，因为数据库必须完全就绪才能创建表

如果 SmartNotes 用户量从 100 增长到 10 万，你会怎么做架构扩展？
杀手级

参考答案

分层扩展策略：

  1. Gateway 层：Nginx 负载均衡多个 Gateway 实例，JWT 验证无状态可水平扩展
  2. Agent 层：每个 Agent 类型部署多实例，Gateway 通过轮询/随机选择转发。QA Agent 可能需要更多实例（知识库查询密集）

  6. LLM API：实现请求队列 + 令牌桶限流，避免 API 配额耗尽。考虑本地部署开源模型（如 Qwen）降低 API 依赖

具体优化：

  • 知识库按用户分 Collection，避免单 Collection 数据量过大
  • LLM 响应 CDN 缓存热点问答
  • 前端静态资源 CDN 加速
  • 加入 Prometheus + Grafana 监控各服务 QPS、延迟、错误率

如何设计一个支持多租户的 SmartNotes？不同学校/机构的数据如何隔离？
困难
▼

参考答案

多租户隔离方案：

  1. 数据库层：MySQL 加入 organization_id 字段，所有查询都带租户过滤条件。使用 Row-Level Security 或应用层过滤
  2. ChromaDB 层：每个租户使用独立的 Collection（如 org_{id}_knowledge），元数据中加入 org_id
  3. Redis 层：Key 前缀加租户标识（如 tenant:{org_id}:rate_limit:{user_id}）
  4. Gateway 层：JWT Payload 中加入 org_id，转发时传递 X-Org-Id Header
  5. 限流策略：按租户配额 + 按用户限流双层控制

隔离级别选择：

  • 共享数据库 + 行级隔离（成本低，适合中小规模）
  • 独立 Schema（中等成本，适合中大规模）
  • 独立数据库实例（成本高，适合大型客户/数据合规要求高的场景）

如果 LLM API 宕机了，你的系统怎么处理？如何设计降级方案？
困难
▼

参考答案

降级策略：

  1. 超时与重试：httpx 设置连接超时 10s + 请求超时 300s，配合指数退避重试（最多 3 次）
  2. 缓存降级：Redis 缓存命中时直接返回，不调用 LLM。热点问答即使 LLM 宕机也能响应
  3. 备用模型：配置备用 LLM（如 qwen 不可用自动切换到 DeepSeek 或本地部署模型）
  4. 优雅错误：捕获异常后返回友好提示（如"AI 服务暂时不可用，请稍后重试"），而非 500 错误
  5. 熔断机制：连续 N 次 LLM 调用失败后，暂时停止调用，直接返回缓存或错误提示，避免雪崩

项目中的现有处理

ai_client.py 中已有 MOCK_RESPONSES 兜底机制——API 调用失败时返回预设的 Mock 回复。虽然简单，但体现了降级意识。
